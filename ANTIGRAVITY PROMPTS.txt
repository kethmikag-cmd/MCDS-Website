-------------------------------------------------------------------------------------------------------------------------------------------------------------Review the entire Firebase Firestore implementation and optimize it for the lowest possible read count.

Project Context:
- School media competition registration system
- Expected registrations: 2500+
- Expected schools: 250+
- Running on Firebase Spark (free tier)
- Priority: minimize Firestore reads while maintaining functionality

Requirements:

1. Registration Form
- Users should only write data to Firestore when submitting.
- Do not read the registrations collection during form submission.
- After successful submission, the registrations recorded message (which was configured and setup in an earlier prompt) is to be displayed without re-querying the firestore database.

2. Registration Collection
- Store registrations in:
  registrations/{autoDocumentId}

Fields:
- fullName
- school
- email
- phone
- category
- submittedAt

3. Homepage Statistics
DO NOT count registrations by reading the entire registrations collection.

Instead create:
stats/overview

Example:
{
  contestantCount: 2500,
  schoolCount: 250,
  lastUpdated: timestamp
}

Homepage counters must read only this single document.

4. Admin Dashboard

- Never load all registrations at once.
- Use pagination.
- Load only 40 registrations per page.
- Implement Next and Previous page controls.
- Display page numbers in a horizontal pagination bar.
- Allow direct navigation by clicking any page number.
- Highlight the currently active page.
- Use Firestore cursor-based pagination.
- Avoid offset-based pagination.
- Do not fetch data for pages the user has not opened.
- Cache previously visited pages during the session to reduce repeated reads.
- Store and reuse pagination cursors for previously visited pages.

5. Search & Filters
- Search by school.
- Search by category.
- Query only matching documents.
- Never download all registrations and filter on the client.

6. Firestore Usage Rules
- Do NOT use onSnapshot() anywhere unless absolutely necessary.
- Prefer getDocs() and getDoc().
- Do NOT poll Firestore repeatedly.
- Avoid unnecessary re-fetching when navigating between pages.
- Cache previously loaded data where appropriate.

7. Static Data
If schools, categories, rules, or competition information rarely change:
- Fetch once.
- Cache locally.
- Do not repeatedly query Firestore.


8. Performance
- Ensure every page minimizes reads.
- Remove duplicate Firestore queries.
- Reuse existing data whenever possible.
- Prevent accidental full-collection reads.

9. Security
Create Firestore security rules so:
- Public users can create registrations.
- Public users cannot read registrations.
- Public users cannot edit registrations.
- Public users cannot delete registrations.
- Admin users retain full access.

10. Final Review
Audit all Firestore code and identify:
- Potential read-heavy queries
- Unnecessary collection scans
- Unnecessary listeners
- Duplicate requests
- Opportunities to reduce reads further

Goal:
Design the application so it can comfortably handle 2500+ registrations on the Firebase Spark free plan with minimal Firestore read consumption.
